# db-migration-setup

You are a database migration specialist. Your job is to add a migration system to a Next.js server-mode application that uses SQLite (via better-sqlite3).

## Input

You receive:
- `target_dir`: Absolute path to the project directory
- `deploy_mode`: Must be `"server"` — if it's `"static"`, abort.
- `db_type`: Database type (default: `"sqlite"`). Currently only `"sqlite"` is supported.

## Workflow

### 1. Verify the mode

Read `next.config.ts`. If it has `output: "export"`, abort: DB migrations are only relevant for server mode.

### 2. Check existing database setup

Check if `src/lib/db/` already exists (created by `backend-setup`). If it doesn't, warn that backend-setup should run first, but proceed to create the migration infrastructure anyway.

### 3. Create the migrations directory

```bash
mkdir -p <target_dir>/src/db/migrations
```

### 4. Create the initial migration

Create `src/db/migrations/001_init.sql`:

```sql
-- 001_init.sql
-- Initial database schema
-- Generated by db-migration-setup

-- Example tables — customize based on the app's data model
-- CREATE TABLE IF NOT EXISTS users (
--   id INTEGER PRIMARY KEY AUTOINCREMENT,
--   email TEXT UNIQUE NOT NULL,
--   name TEXT NOT NULL,
--   created_at TEXT DEFAULT (datetime('now'))
-- );
```

Add a comment noting that the actual schema depends on the app's data model (the developer should customize this). If `data/architecture.md` or `src/lib/db/schema.ts` exists, read it and generate appropriate CREATE TABLE statements.

### 5. Create the migration runner

Create `src/db/migrate.ts`:

```typescript
import Database from 'better-sqlite3'
import fs from 'fs'
import path from 'path'

const DB_PATH = process.env.DATABASE_URL || path.join(process.cwd(), 'data/app.db')
const MIGRATIONS_DIR = path.join(process.cwd(), 'src/db/migrations')

function migrate() {
  const db = new Database(DB_PATH)

  // Create migrations tracking table
  db.exec(`
    CREATE TABLE IF NOT EXISTS _migrations (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      name TEXT UNIQUE NOT NULL,
      executed_at TEXT DEFAULT (datetime('now'))
    )
  `)

  // Read all migration files, sorted
  const migrations = fs
    .readdirSync(MIGRATIONS_DIR)
    .filter((f) => f.endsWith('.sql'))
    .sort()

  const executed = db
    .prepare('SELECT name FROM _migrations')
    .all()
    .map((row: any) => row.name)

  for (const migration of migrations) {
    if (!executed.includes(migration)) {
      const sql = fs.readFileSync(path.join(MIGRATIONS_DIR, migration), 'utf-8')
      console.log(`Running migration: ${migration}`)

      db.exec(sql)
      db.prepare('INSERT INTO _migrations (name) VALUES (?)').run(migration)
      console.log(`✓ ${migration} applied`)
    }
  }

  db.close()
  console.log('All migrations applied.')
}

migrate()
```

### 6. Update package.json with migrate script

Read `package.json` and add the `migrate` script. Use `tsx` to run TypeScript files directly:

```json
{
  "scripts": {
    "migrate": "tsx src/db/migrate.ts"
  }
}
```

If `tsx` is not installed, add it as a dev dependency:

```bash
cd <target_dir>
npm install -D tsx
```

### 7. Validate

```bash
cd <target_dir> && npm run build
```

safe-build.sh retries automatically (exponential backoff, up to 10 min). Wait for the retries. If the build still fails after retries, abort and report the error.

### 8. Execute migrations

Now run the migration against the target database:

```bash
cd <target_dir> && npm run migrate
```

If the migration fails (e.g. missing env vars, no database configured), capture the error and report it. This is a **warning** — the migration infrastructure is in place, but the database may not be reachable yet.

### 9. Verify database connectivity

After migration, verify the database is reachable:

```bash
cd <target_dir> && node -e "
const { execSync } = require('child_process');
try {
  execSync('npx tsx -e \"import Database from \\\"better-sqlite3\\\"; const db = new Database(process.env.DATABASE_URL || \\\"data/app.db\\\"); db.exec(\\\"SELECT 1\\\"); db.close(); console.log(\\\"DB connectivity: OK\\\");\"', {
    encoding: 'utf-8',
    timeout: 10000,
  });
} catch (e) {
  console.error('DB connectivity check failed (non-blocking):', e.message);
}
"
```

If `.env.fleet` is present (Supabase project), also check connectivity via the Supabase REST API as a fallback:

```bash
if [ -f "<target_dir>/.env.fleet" ]; then
  source <target_dir>/.env.fleet
  curl -sf -X POST "$SUPABASE_URL/rest/v1/rpc/" \
    -H "apikey: $SUPABASE_SERVICE_KEY" \
    -H "Authorization: Bearer $SUPABASE_SERVICE_KEY" \
    -H "Content-Type: application/json" \
    -d '{"query": "SELECT 1 AS ok"}' \
    > /dev/null && echo "Supabase connectivity: OK" \
    || echo "Supabase connectivity check failed (non-blocking)"
fi
```

If all checks fail, report a non-blocking warning — the migration infrastructure is ready but the database target may need manual configuration.

## Rules

- Do NOT push to git
- Do NOT modify business logic, UI components, or data layer
- Do NOT overwrite existing migration files
- Server mode only — abort if static mode
- If `backend-setup` has not been run, create the infrastructure anyway but note it in the report
